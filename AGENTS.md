# AGENTS.md — fluxo-kmp-conf

This file targets **maintainers/contributors** of this Gradle plugin (`io.github.fluxo-kt.fluxo-kmp-conf`), not consumers. Consumer docs are in `README.md`.

The plugin lazily, target-aware, configures Kotlin/JVM, Android, KMP, Compose, IDEA-plugin, and Gradle-plugin modules from one root entry point. Built **with itself** via `includeBuild("self")`.

> Read the **Surprises** section first — most footguns here aren't grep-able.

## Vibe / core principles
- **Lazy on-demand**: nothing configured eagerly; modules with every target disabled must still configure cleanly. Prefer `tasks.named` / `withType` over eager creation.
- **Per-target filtering**: every target gate honours `KMP_TARGETS` (CSV of codes from `KmpTargetCode.kt`) or `KMP_TARGETS_ALL=true`. Don't assume a target is present.
- **Opt-in experimental**: defaults are conservative — check the relevant `FluxoConfigurationExtension*` KDoc and `CHANGELOG.md` before flipping any default.
- **Property/env flags only via `PropsAndEnv.kt`** (`envOrPropFlag` / `envOrPropValue`); for shelling out, the same file's `runCommand` helper. Never `System.getenv` / raw `Runtime.exec`.
- **Configuration-cache compatible**: no `Project` access at execution time, no eager task creation. CC is on, `problems=warn` — silent violations are possible, treat warnings as errors during review.
- **Public API is the contract**: KDoc on `fluxo/conf/dsl/FluxoConfigurationExtension*.kt` is authoritative. New flags need impl in `FluxoConfigurationExtensionImpl*` plus parent-inheritance fallback where the KDoc says so.
- **`kotlin.internal` annotations are fair game**: `@LowPriorityInOverloadResolution`, `@kotlin.internal.OnlyInputTypes`, and similar `kotlin.internal` annotations provide real semantic value (overload-resolution control, type-safety enforcement). The `kotlin.internal` package name does NOT mean "forbidden to consumers"; it means "no public stability promise". Using them with `@Suppress("INVISIBLE_MEMBER", "INVISIBLE_REFERENCE")` is intentional policy here. If they break in a future Kotlin version, fix it then. **Never silently remove them to avoid a compiler warning.** Kotlin 2.2 emits a meta-warning when such suppression is used ("behavior UNSPECIFIED"); that meta-warning is itself accepted as a known cost of the more expressive API.
- **`@Suppress("INVISIBLE_MEMBER"/"INVISIBLE_REFERENCE")` is often PREFERABLE to reflection.** Replacing a direct typed access with `javaClass.getMethod(...)` / `getDeclaredField(...)` reflection is a quality DEGRADATION: reflection is slower, string-coupled, invisible to the IDE, defeats refactoring, and fails at runtime instead of compile time. Replace `@Suppress` accesses ONLY when the alternative is genuinely superior across ALL dimensions — correctness, type safety, DX, maintainability, performance. Reflection is only valid when: (a) the target might NOT be present at runtime across the supported version range (layer-2 compat), or (b) multiple incompatible variants of the API must be supported simultaneously and cannot be compiled together. **Removing a `@Suppress` to silence a warning and replacing it with reflection is not an improvement — it is a regression.**
  - Corollary: Kotlin's diagnostic model treats an `internal` property/member as BOTH an inaccessible member (`INVISIBLE_MEMBER`) AND an invisible symbol (`INVISIBLE_REFERENCE`), even when its type is public. Practically: accessing any `internal` member from outside its module requires BOTH suppressions regardless of whether the class itself is public or internal.

## Supported version matrix (READ BEFORE assessing dead code or compat fallbacks)

When reasoning about API drift, "dead-under-our-floor" code, ProGuard keep rules, or reflective lookups, distinguish three layers — they're NOT the same range. Past mistakes (e.g., calling a `LintTool.getVersion()` keep rule "dead under AGP 9 floor") came from evaluating against layer 1 only.

- **Build pin (layer 1)** — what `gradle/libs.versions.toml` PINS, what `:plugin:compileKotlin` resolves against. Source of truth: `libs.versions.toml`. Currently: Kotlin 2.2.21, KSP 2.2.21-2.0.5, Gradle wrapper 9.3.1, AGP 9.1.1 (compileOnly), Compose Multiplatform 1.10.3, Spotless 8.4.0, Detekt 1.23.8, IntelliJ Platform 2.15.0, BCV 0.18.1, Vanniktech 0.36.0. AGP being `compileOnly` means it does **not** ship with the published artefact — consumers bring their own AGP.
- **Consumer minimum (layer 2)** — what 0.14.0+ promises consumers can run against. Sources of truth: README "Targeted for …" line + CHANGELOG breaking-changes section. **Both AGP 8.x and AGP 9.x are first-class supported (not "8 legacy, 9 primary" — both are equal-citizen runtime targets).** Kotlin language version **2.0** is the absolute minimum supported (consumer KGP 2.0.0+); the catalog default for `kotlinLangVersion` is **2.1**, and that's what README/CHANGELOG promise without configuration. Layer-3 ceiling is unbounded — Kotlin 2.1+ through the latest betas/RCs is supported. Consumer overrides to `kotlinLangVersion = 2.0` are valid but reduced-feature; overrides below 2.0 (1.9 / 1.8 / earlier) are NOT supported and the corresponding code paths have been dropped. Gradle **9.0+** daemon (Gradle 8.x daemons embed only Kotlin 2.0.x and can't load our Kotlin 2.2.x classes). AGP **8.0+** for the non-KMP `com.android.library` path (`setupAndroidCommon` body uses `CommonExtension` with reflective pre-8.2 `LintTool.getVersion()` fallback at `SetupAndroidLint.kt:301`). KMP+Android is dual-line and auto-routed: AGP **8.0–8.x** via the legacy `com.android.library` + `kotlin("multiplatform")` co-application path; AGP **9.0+** via the new `com.android.kotlin.multiplatform.library` plugin id (REQUIRED — AGP 9 hard-rejects co-application). Detection lives in `AgpVersion`; routing in `TargetAndroidContainer.Library`.
- **Consumer maximum (layer 3)** — forward-compat range. **Unbounded** by design; the structural defences are: `noSuchMethodSafe { }` (`fluxo.conf.impl.android.AndroidForwardCompat`) at every AGP setter site; `SetupKotlinCompatibility` self-warns when consumer Kotlin exceeds the known compatibility table; Detekt language-version clamp self-warns when consumer Kotlin exceeds Detekt's supported max. New consumer-facing surfaces MUST add equivalent defences before adding setter calls.

**Compat-audit principle (load-bearing — fail this and you ship regressions)**: when assessing "is X dead?" or "can we drop branch Y?", evaluate against **layer 2 (consumer minimum)** OR specifically the layer-2 sub-range that activates the code (e.g., reflective fallbacks have their own narrower band — pre-8.2 AGP, pre-2.0 Kotlin). NEVER evaluate against layer 1 alone — a reflective fallback or keep rule that's a no-op under our pinned-AGP shrink classpath is STILL load-bearing if any consumer-supported version in layer 2 exposes the symbol differently. Audit targets where this principle applies: ProGuard/R8 keep rules in `pg/keep.pro` (and `pg/keep-api-autogenerated.pro`); every `Class.forName` / `getDeclaredMethod` lookup; version-branched paths in `SetupKotlinOptions.kt`, `SetupAndroid.kt`, `KmpConfigurationContainerDslImpl.kt`. When a branch IS genuinely dead across **all** of layer 2, the cleanup is a deliberate ABI/CHANGELOG event (raise the layer-2 floor first via README + CHANGELOG, then drop the branch).

## Layout (only the non-obvious)
- `fluxo-kmp-conf/` — published plugin. Public DSL: top-level `FkcSetup*.kt` (all share `@JvmName("Fkc") @JvmMultifileClass`). Engine: `fluxo/conf/`. Public API dump: `api/plugin.api`. Verification baselines: `detekt-baseline.xml`, `lint-baseline.xml`. Generated `BuildConstants` → `fluxo.conf.data`.
- `self/` — sibling module sharing sources with `fluxo-kmp-conf/` via `kotlin.srcDir("../fluxo-kmp-conf/src/main/kotlin")` and `includeBuild`'d so the plugin self-applies. `buildSrc` is intentionally avoided.
- `checks/{kmp,gradle-plugin,compose-desktop}/` — three integration-test Gradle builds with their own `settings.gradle.kts` and `build.gradle.kts`, but `gradle/`, `gradlew`, `gradlew.bat`, `config/` are **symlinks to root** (one wrapper, one catalog). Two of the three (`gradle-plugin`, `compose-desktop`) additionally symlink `gradle.properties` to `self/gradle.properties` (CC=false). CI cd's into each.
- `gradle/libs.versions.toml` — single source of truth for versions; resource-copied at build time as `fluxo.versions.toml` (comments stripped) and exposed at runtime via `FluxoVersionCatalog`.
- `ROADMAP.md` = research/links scratchpad, **not** active TODOs. `CHANGELOG.md` follows Common Changelog.
- ProGuard/R8 keep rules live per-module in `<module>/pg/` (plugin's own at `fluxo-kmp-conf/pg/`).
- Optional integrations live in `fluxo/conf/feat/` (`Setup*.kt` / `Prepare*.kt` patterns); per-target KMP containers in `fluxo/conf/dsl/container/{,impl,target}`.

## Things that will surprise you (append new finds; prune outdated)
- **`self/build.gradle.kts` and `fluxo-kmp-conf/build.gradle.kts` carry mirrored `dependencies {}` and `buildConfig {}` blocks** because `:self` re-uses the plugin's main sources via `kotlin.srcDir(...)` and must declare the same buildscript classpath. The mirrored regions are delimited by `// MIRROR-START` / `// MIRROR-END` markers and a byte-identity invariant is enforced by the `verifyBuildScriptMirror` task on `:plugin:check`. Update both regions in lockstep; CI fails the build on drift.
- **`kotlin-stdlib` is explicitly excluded** from `implementation` in the plugin's build (`exclude(group = "org.jetbrains.kotlin", module = "kotlin-stdlib*")`) plus `kotlin.stdlib.default.dependency=false`. Gradle injects its own stdlib at runtime; bundling ours triggers ClassLoader conflicts. **Don't "clean up" that block.**
- **Kotlin lang/api are pinned** in `libs.versions.toml` to a version older than the build Kotlin — this is the consumer-compatibility surface. Bumping is a breaking-change decision, not a routine bump.
- **Module rename trap**: `settings.gradle.kts` renames `:fluxo-kmp-conf` → `:plugin`. The string `plugin` is referenced by `jitpack.yml`; renaming breaks JitPack snapshot consumers (comment on the line warns).
- **Git symlinks required for development** (Windows: enable per the README link). Without them the shared-sources trick fails. `.aiexclude` is itself a symlink to `.cursorignore`.
- `org.gradle.vfs.watch=false` and `org.gradle.unsafe.watch-fs=false` are intentional — VFS watching errors with "same file via different paths" because of source sharing. Don't re-enable.
- `MAX_DEBUG=true` is a **developer dogfooding flag** committed in `self/gradle.properties` and `checks/*` builds so plugin development runs with full verbose output. It is intentionally absent from root `gradle.properties` (consumer-facing reference). External contributors can also set it via local `gradle.properties` or env var. `FLUXO_VERBOSE` is a lighter variant. Both are read via `envOrPropFlag` in `PropsAndEnv.kt`.
- **Tests auto-disable** when `DISABLE_TESTS=true`, the build is an *included* build with no startup tasks, or `check`/`test` are excluded. If `prepareKotlinIdeaImport` runs but expected tasks don't, see `FluxoKmpConfContext.testsDisabled`.
- **IDE-sync-only config** uses `onProjectInSyncRun` (detected via `prepareKotlinIdeaImport` / `prepareKotlinBuildScriptModel` task names). Use it instead of `afterEvaluate` for IDE-import hooks.
- The plugin **applies only to root projects** (`checkIsRootProject`) and **requires a Kotlin plugin already on the classpath** before it's applied — order in `plugins {}` matters.
- Shrinker `replaceOutgoingJar = false` in the plugin's own build is intentional — chained R8/ProGuard tasks are *verification-only*, they don't replace the published jar.
- `afterEvaluate` is used sparingly and always carries a FIXME — prefer lazy `Provider` / `NamedDomainObjectContainer` APIs in new code.
- Conventional Commits are **convention only** — not CI-enforced. PR titles + commit messages rely on reviewer discipline.
- **Detekt's `CyclomaticComplexMethod` counts local-function complexity into the enclosing method's score.** Extracting a helper into a local `fun` does NOT reduce the host's CC. To shrink it, hoist the helper to a top-level extension (see the `applyReproducibleArchivePermissions` extraction).
- **Pure-Kotlin symbols that need unit tests must live in a file with zero KGP refs.** The Kotlin file-class `<clinit>` resolves every top-level symbol declared in the file, so a single `org.jetbrains.kotlin.gradle.*` reference (those types are `compileOnly`) explodes the file's tests with `NoClassDefFoundError`. Pattern: split the testable surface to its own file (e.g. `KotlinVersionTable.kt`, `DetektKotlinClamp.kt`); the file-level docstring documents the discipline.
- **`kotlin.KotlinVersion`'s `compareTo` is patch-inclusive.** `KotlinVersion(2, 0, 1) > KotlinVersion(2, 0, 0)` is `true` — be deliberate about whether you mean "later patch of the same minor" or "later minor". For minor-bracket membership use a `FIRST_UNTABULATED_<...>` constant compared with `>=`, not a `LATEST_TABULATED_<...>` compared with `>` (the latter false-fires on every patch above 0).
- **Never use `String.toFloat()` for version comparison.** `"1.10".toFloat() == 1.1f` ranks falsely below `1.9`; `"2.10".toFloat() == 2.1f` ranks falsely equal to `2.1`. Parse to `kotlin.KotlinVersion` (no KGP coupling) and compare lexicographically — see `parseDetektLangVersion`.
- **AGP 9's `LibraryExtensionImpl` no longer implements legacy `com.android.build.gradle.TestedExtension`.** The legacy `TestedExtension` interface still exists on the AGP-9 classpath (so source-level references compile), but the runtime extension instance returned from `extensions.findByName("android")` is no longer castable to it — code doing `<TestedExtension>` lookups crashes with `ClassCastException` at the first method call. Use `com.android.build.api.dsl.CommonExtension` (modern, parents both `LibraryExtension` and `ApplicationExtension` across AGP 8 and 9) instead. Beware: AGP-9 dropped the action-form `compileOptions { }` helper from `CommonExtension` — only the getter survives, so configure via the property directly (`compileOptions.sourceCompatibility = …`). `CommonExtension.buildTypes` is also covariant-wildcarded (`NamedDomainObjectContainer<? extends BuildType>`) so `maybeCreate` / `add` don't compile against it — for those, specialise on the concrete subtype (`LibraryExtension` / `ApplicationExtension` modern).
- **`setupAndroidCommon` was migrated to `CommonExtension` in 0.14.0** so it works on AGP 8 and AGP 9 from the same body. Three caveats baked into its signature: (1) `defaultConfig { }` Action-form lives only on the concrete subtypes, so the shared body uses `defaultConfig.apply { }`; (2) `targetSdk` was removed from `LibraryBaseFlavor` in AGP 9 and is dispatched through an `is ApplicationExtension` smart-cast helper; (3) `buildTypes` is covariant-wildcarded on `CommonExtension`, so the `maybeCreate(DEBUG)` / `maybeCreate(RELEASE)` calls dispatch via `when (this) { is LibraryExtension -> … is ApplicationExtension -> … }`. Non-KMP coverage lives in `checks/android/`; AGP-8 KMP+Android coverage is still a gap (sibling check or build-property toggle in `checks/kmp/` would close it).
- **`kotlin-compiler-embeddable` and `detekt-core` are `compileOnly`, never `implementation`.** KGP 2.1+ no longer drags `kotlin-compiler-embeddable` in transitively, and KGP runtime warns on its presence alongside the Gradle plugin. Putting it on the published runtime classpath (i.e. `implementation`) leaks it onto the consumer's buildscript classpath; Gradle's `<latest>` resolution then upgrades it past whatever KGP version the consumer uses, and inlined `BuildPerformanceMetrics.add$default` / similar helpers blow up with `NoSuchMethodError` from `GradleCompilationResults` on the very first `compileKotlin`. Symptom is most visible on AGP 9 + non-KMP (AGP ships its own KGP at 2.2.10 inside 9.1.1; we pin 2.2.21). Compile-time access is fine — `KotlinToolingVersion` is in `kotlin-tooling-core` (provided by KGP transitively) and `detekt-tooling-api.BaselineProvider` is provided by `detekt-gradle-plugin`. The dependencyGuard baselines (root + every `checks/*`) are the structural enforcer: re-introducing the leak fails `:plugin:check`.
- **AGP setter call sites must be wrapped in `noSuchMethodSafe { }`** (`fluxo.conf.impl.android.AndroidForwardCompat`). The plugin compiles against one catalog-pinned AGP but is *applied* against whatever AGP the consumer brings, and AGP routinely deprecates/removes individual setters across minor versions (`targetSdk` removed from `LibraryBaseFlavor` in AGP 9 is the canonical precedent; `*Preview` setters are `@Incubating` with no stability promise). The helper absorbs `NoSuchMethodError` *only* — real bugs (NPE/ClassCast/type mismatch from consumer config) propagate. Wrap **per setter** (or per single-purpose helper like `applyAgpSdkProperty` / `applyCompileSdk`), not at the function level, so one removed property doesn't skip a sibling property's assignment. Used by both `setupAndroidCommon` (legacy) and `setupKmpAndroidExtension` (AGP-9 KMP). The two non-Android NSME catches (`DomainObjectUtils.namedLazy`, `SetupKotlinCompatibility.KOTLIN_LANG_VERSION`) deliberately stay site-local — they compute *real fallback values*, not "skip silently", so the value-producing shape doesn't fit the helper.
- **`MergeDetektBaselinesTask` parses Detekt baseline XML directly — never reflectively loads Detekt types.** Each Gradle plugin runs in an isolated ClassLoader scoped to its own POM runtime dependencies. `Class.forName("io.github.detekt.tooling.api.BaselineProvider")` succeeds at *test* time (flat classpath) but throws `ClassNotFoundException` at *task-execution* time because the calling classloader is our plugin's loader, not Detekt's isolated loader. There is no cross-classloader bridge. The correct approach: parse the baseline as XML via `DocumentBuilderFactory` (bootstrap classloader, always visible). The Detekt baseline format (`<SmellBaseline>/<ManuallySuppressedIssues>/<CurrentIssues>/<ID>`) is a stable, documented contract. `MergeDetektBaselinesReflectiveTest` falsifies the XML merge invariants without any Detekt type references.
- **AGP 8 vs AGP 9 KMP+Android plugin id is a hard split, not a version bump.** Legacy `com.android.library` + `kotlin("multiplatform")` co-application is hard-rejected by AGP 9; the AGP-9 KMP-aware `com.android.kotlin.multiplatform.library` plugin auto-creates the `android` KMP target via `kotlin { android { } }` and uses the disjoint `KotlinMultiplatformAndroidLibraryExtension` / `KotlinMultiplatformAndroidLibraryTarget` types (do NOT extend `LibraryExtension` / `KotlinAndroidTarget`). Detect the line with `AgpVersion.isAgp9OrLater(project)`. Routing in this codebase: `setupKmpAndroidExtension` (namespace/compileSdk/minSdk slot-fill, wired in `setupKotlin`) + `setupKmpAndroidLint` (Lint config, wired in `setupVerification`) handle the new path; the legacy `setupAndroidCommon` / `setupAndroidLint` handle the old. `TargetAndroidContainer.Library` runs both via the virtual `needsLegacyTargetCreation` property — under AGP 9 it skips `androidTarget()` (the auto-created target collides) and warns if `lazyAndroid` is non-empty (lambdas typed for the disjoint `LibraryExtension` cannot run on the new extension). `androidApp { }` fail-fasts under AGP 9: there is no KMP-aware AGP application plugin upstream.
- **Kotlin 2.2 K2 IR optimizer crashes when `@Suppress("INVISIBLE_*")` and an `internal const val` getter coexist in the same compilation unit.** `IrConstOnlyNecessaryTransformer` / `IrConstEvaluationContext` scans the ENTIRE compilation unit, not just the annotated function. So even when the `@Suppress` is on a private helper at the bottom of the file, ANY `internal const val` getter call elsewhere in the same file (even from a different package via same-module visibility) crashes with `AssertionError: Error occurred while optimizing an expression: CALL 'internal final fun <get-CONST_NAME>(): ...`. Workaround: wherever a file must use `@Suppress("INVISIBLE_*")`, never call `internal const val` getters from other files — use the literal value instead. Document the literal with a comment citing this bug.

## Common tasks
| Task | Command |
|---|---|
| Build & full check (root plugin) | `./gradlew build assemble check` |
| Run an integration check | `cd checks/<kmp\|gradle-plugin\|compose-desktop> && ./gradlew build check` |
| Update **all** baselines (api, detekt, lint, depGuard) | `./updateBaseline` — runs root + every `checks/*` |
| Per-area baselines | `./gradlew apiDump dependencyGuardBaseline detektBaselineMerge updateLintBaseline --continue` |
| Limit KMP targets | `KMP_TARGETS=<csv> ./gradlew …` (codes in `KmpTargetCode.kt`) or `KMP_TARGETS_ALL=true` |
| Disable shrinker | `DISABLE_R8=true …` (aliases in `PropsAndEnv.kt`) |
| Force release semantics | `RELEASE=true …` |

Tests are integration-style: `fluxo-kmp-conf/src/test/` is sparse (mostly shrinker via `kotlin-compile-testing` + ProGuard/R8). The real test surface is the three `checks/*` builds.

## Conventions
- **Conventional Commits** required (`feat(scope): …`); types and rules in `CONTRIBUTING.md`. PR titles follow the same. Merges are **`--ff-only`** (enforced by `pr-fast-forward.yml`); avoid merge commits except on hotfix branches.
- Public API: anything new needs `./gradlew apiDump` + CHANGELOG entry. Internal-only: annotate `@fluxo.annotation.InternalFluxoApi` (wired into BCV `nonPublicMarkers`).
- **CHANGELOG.md is strictly consumer-facing.** Each entry must answer *"what changes in the consumer's build?"*: new/changed/removed public DSL or ABI; new behaviours/warnings the plugin emits; consumer-compat-floor bumps; wrapped-plugin bumps consumers can observe; bug fixes consumers hit; doc that teaches a previously-undocumented consumer contract. **NOT**: CI, baselines, `verifyBuildScriptMirror`, internal refactors/renames/file-splits, test seeds, AGENTS/ROADMAP edits, regenerated `api/*.api`, SHA-pin / Dependabot grouping, internal-package decoupling that doesn't ship today. Failing the test → commit message, not CHANGELOG.
- Editorconfig: 4-space Kotlin / `.kts`, 100-col line length. ktlint-official enabled; `trailing-comma` and `no-consecutive-blank-lines` rules disabled.
- Diffs stay surgical: don't reformat untouched files, don't refactor during bug fixes.
- **Bumping the plugin's own version**: update `gradle/libs.versions.toml` (`version`) and the `README.md` example block (consumer-visible Gradle Plugin Portal version).
- **Adding a 3rd-party Gradle plugin we wrap**: mirror in `self/build.gradle.kts` *and* `fluxo-kmp-conf/build.gradle.kts` (`compileOnly`/`implementation` + `buildConfigField`). See *Surprises* — no guard.
- Release: `vanniktech-mvn-publish` + `release.yml`; SCM derived from `SCM_TAG` / `BUILD_NUMBER` env (see `PropsAndEnv.kt`).

## When something surprises you
If you encounter behaviour, layout, config, or tooling that genuinely surprised you (hidden coupling, non-obvious flag, subtle CI requirement, footgun), **flag it to the user** and **append a one-liner to "Things that will surprise you"** so the next agent doesn't repeat the discovery cost.

Before appending: **scan the existing list and prune anything no longer true** (resolved FIXMEs, removed flags, fixed gotchas). The list must reflect *current* state, not history. When you fix or remove a coupling, prune the corresponding entry in the same change.
